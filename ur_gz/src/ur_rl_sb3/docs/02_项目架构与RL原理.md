# 项目架构与 RL 原理（面向实战）

---

## 1. 整体架构图（文字版）

`ur_rl_sb3`（你写）

- `launch/ur_rl_sim.launch.py`：只负责 include 官方仿真 launch，并指定控制器
- `joint_goal_env.py`：Gym 环境封装
- `train_sac.py` / `eval_sac.py`：训练与评估入口

`ur_simulation_gz`（官方）

- xacro 展开机器人模型
- 启动 Gazebo
- 启动 `gz_ros2_control`
- 启动 `controller_manager` 和相关 controller
- `/clock` 桥接

---

## 2. 数据流：每一步发生了什么

每次 `env.step(action)`：

1. RL 算法输出 6 维动作（范围 -1 到 1）
2. 环境把动作映射成关节命令增量（`action_scale`）
3. 发布到 `/forward_position_controller/commands`
4. Gazebo + ros2_control 推动机械臂运动
5. 环境读取 `/joint_states` + TF 末端位姿
6. 计算误差、奖励、是否结束
7. 返回给 SB3

---

## 3. 观测、动作、奖励怎么设计的

## 3.1 观测（Observation）

当前实现是 18 维：

- 关节位置 6 维
- 关节速度 6 维
- 末端位置 3 维
- 末端到目标误差 3 维

这样做的好处：

- 既给了机器人本体状态，也给了任务信息（目标差值）

## 3.2 动作（Action）

- 连续 6 维动作，范围 `[-1, 1]`
- 通过 `action_scale` 映射为“每步关节增量”
- 再做关节限幅，防止出界

## 3.3 奖励（Reward）

- 主项：`-误差范数`（距离越小越好）
- 正则：`-0.01 * 动作范数`（抑制过激动作）
- 成功加分：达到阈值后额外 +5

## 3.4 终止条件

- `goal_reached`：误差 < `goal_tolerance`
- `max_steps`：达到最大步数还没成功

## 4. 为什么在 RViz 能看见目标点？

环境 reset 时会发布 `visualization_msgs/Marker`：

- topic：`/ur_rl_sb3/goal_marker`
- frame：`base_link`
- 形状：红色球体

你在 RViz 添加 `Marker` display，选这个 topic，就能看到每回合随机目标点。

## 5. 为什么这个项目“简历价值高”

因为它同时覆盖：

- 机器人软件工程（ROS2 launch/topic/TF/控制链路）
- 强化学习工程（环境封装、奖励设计、训练/评估闭环）
- 可观测性与可调试（可视化 marker、回合结束日志）

这比“只跑了一个 notebook”更像真实工程。
